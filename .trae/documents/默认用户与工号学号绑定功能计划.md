# 默认用户与工号/学号绑定功能实施计划

## 一、需求分析

### 1.1 功能需求

1. **默认用户数据**：系统预置默认用户信息（工号/学号、姓名、用户类型）
2. **首次登录绑定**：用户首次微信登录后，需要绑定工号/学号
3. **绑定验证**：验证工号/学号是否存在于预置用户列表中

### 1.2 现有系统分析

* 用户表已有 `student_no` 字段（学号/工号）

* 用户类型：学生(1)、教师(2)、外部人员(3)

* 当前登录：微信OAuth自动创建用户，`isNewUser` 根据 `name == null` 判断

* 前端已有用户资料更新接口

***

## 二、技术方案

### 2.1 数据库设计

#### 新增字段：用户表增加绑定状态

```sql
ALTER TABLE user ADD COLUMN is_bound TINYINT DEFAULT 0 COMMENT '是否已绑定工号/学号: 0-未绑定, 1-已绑定';
ALTER TABLE user ADD COLUMN bound_at DATETIME COMMENT '绑定时间';
```

#### 新增表：预置用户信息表（可选方案）

或直接在用户表中预置数据，openid 为空表示待绑定。

**采用方案**：在用户表预置数据，openid 为 NULL 表示待绑定的预置用户。

### 2.2 后端改造

#### 2.2.1 登录响应增强

`LoginResponse` 新增字段：

* `isBound`: 是否已绑定工号/学号

* `needBind`: 是否需要绑定（新用户且未绑定）

#### 2.2.2 新增绑定接口

```
POST /user/bind
Request: { studentNo: string, name: string }
Response: UserResponse
```

绑定逻辑：

1. 验证工号/学号是否存在于预置用户中
2. 验证姓名是否匹配（可选）
3. 将预置用户信息合并到当前微信用户
4. 删除预置用户记录（或标记为已绑定）

#### 2.2.3 用户资料更新接口增强

允许更新工号/学号（仅限未绑定用户）

### 2.3 前端改造

#### 2.3.1 登录流程增强

* 登录成功后检查 `needBind` 字段

* 如果需要绑定，跳转到绑定页面

#### 2.3.2 新增绑定页面

* 输入工号/学号

* 输入姓名（验证用）

* 提交绑定

## 六、注意事项

1. **数据迁移**：现有用户的 `is_bound` 默认为 0，需要考虑是否批量更新
2. **唯一性约束**：工号/学号在已绑定用户中必须唯一
3. **外部人员处理**：外部人员可能没有工号/学号，允许跳过绑定
4. **安全性**：绑定接口需要验证用户身份，防止恶意绑定
5. **用户体验**：绑定页面支持跳过，后续可在个人中心补绑

***
