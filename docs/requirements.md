# 校园多球馆智能预约系统（V1）- 需求文档

## 1. 背景与目标
本项目是一个校园多球馆、多场地的预约系统，包含：
- 微信小程序端（uni-app）：学生/教师进行预约、取消、查看记录，并出示核销二维码。
- Web 后台管理端（Vue 3）：管理员与场馆员进行资源配置、核销签到、数据统计。
- 后端（Spring Boot 3）：提供统一 API、预约规则校验、数据存储与审计。

V1 目标：
- 减少人工排场，避免重复占用（杜绝“撞场”）。
- 规则标准化（可预约窗口、个人限额、取消规则、违约限制）。
- 提供现场二维码核销与基础运营看板。

V1 不做：
- 支付/押金/退款。
- 多校区/多学校的复杂多租户隔离（V1 以“单校区多球馆”为主）。

## 2. 范围定义
### 2.1 端与技术边界
- 后端：Spring Boot 3（REST API）。
- 小程序：uni-app（微信小程序构建）。
- 后台：Vue 3 Web。

### 2.2 资源模型
- 校区：V1 默认单校区。
- 球馆（Venue）：多个（例如羽毛球馆、篮球馆、乒乓球室）。
- 场地/场号（Court）：隶属某个球馆（例如 1-12 号场、A/B 场）。

### 2.3 时间模型
- 固定时段切片（推荐）：每个球馆可配置 `slot_minutes`（默认 60 分钟，可选 30/90）。
- 一次预约占用 1 个或多个连续时段。

## 3. 角色与权限
V1 角色：
- 学生/教师（小程序端用户）
  - 浏览球馆/场地与可用时段
  - 创建/取消预约
  - 查看个人预约记录
  - 展示核销二维码
- 场馆员
  - 查看今日排场（按球馆/按场地）
  - 扫码核销（签到）
  - 异常处理：标记爽约、手动核销（可选）、临时停用某场地（可选）
- 管理员
  - 球馆/场地管理
  - 预约规则配置
  - 用户与角色管理（RBAC）
  - 统计与审计

权限粒度：
- V1 支持固定角色的 RBAC；可扩展“场馆员只管理其被分配球馆”的数据范围。

## 4. 核心流程
### 4.1 学生/教师（小程序）
1) 微信登录。
2) 选择球馆 -> 场地 -> 日期。
3) 选择可用连续时段。
4) 提交预约。
5) 系统自动确认（规则校验通过即占用资源并生成预约单）。
6) 到场后打开预约单展示二维码。

### 4.2 场馆员（后台）
1) 登录。
2) 打开“今日排场”并按球馆筛选。
3) 扫描用户二维码。
4) 系统校验通过并完成核销签到。

### 4.3 管理员（后台）
1) 配置球馆/场地。
2) 配置规则（时段长度、营业时间、可预约窗口、限额、取消截止）。
3) 查看统计与违约情况。

## 5. 功能需求
### 5.1 登录与账号
- 小程序端：微信登录（`code` -> 后端 -> `openid`）。
- 后台端：账号密码或员工账号体系登录（实现可选），基于 JWT。
- 用户状态：ACTIVE / DISABLED。

### 5.2 球馆/场地管理（管理员）
- 球馆 CRUD：名称、位置、开放日、营业时间、时段长度、可预约窗口等。
- 场地 CRUD：所属球馆、场地名称/编号、运动类型、状态（启用/停用）。

### 5.3 可用性查询与时段生成
- 输入（球馆/场地、日期），返回时段列表：
  - `start/end`
  - 可用状态（free/occupied）
  - 根据角色控制可见信息（普通用户只看到“被占用/可用”；场馆员/管理员可看到占用人信息）。

### 5.3.1 智能推荐（提升预约成功率，V1 重点）
目标：减少用户反复试错，提高一次提交成功率。

推荐触发点：
- 用户选择了运动类型/球馆/日期/期望时段范围（或期望开始时间 + 时长）后。

推荐输出：
- 返回 Top N 个“可预约方案”，每个方案包含：
  - venueId / courtId
  - date, startTime, endTime
  - 方案标签（例如：同馆同场地号相近、时间相近、替代球馆）
  - 推荐理由（简短可解释）

推荐基本策略（V1 规则驱动）：
- 同球馆优先：优先同一球馆内不同场地的可用连续时段。
- 时间相近优先：优先满足期望时间，允许在可配置范围内做 +/- K 个时段的偏移。
- 场地相近优先：如同球馆内有场地号，优先编号相近或同区域。
- 连续性优先：优先满足用户期望的连续时长（例如 2 个 60min 时段）。
- 替代球馆：当同球馆不足时，推荐同运动类型的其他球馆。

可配置项（建议）：
- 推荐方案数 N（默认 10）
- 时间偏移范围 K（默认 2 个时段）
- 是否允许替代球馆（默认允许）
- 连续时长偏好（默认严格满足；可选允许降级）

### 5.4 创建预约（自动确认）
规则校验（V1，按球馆可配置）：
- 时间合法性：在营业时间内；对齐时段；必须连续。
- 可预约窗口：可提前 N 天预约；不可预约过期日期。
- 冲突校验：同一场地同一日期不得存在重叠预约。
- 限额：
  - 个人日限额（例如每天最多 X 个时段/最多 Y 小时）
  - 个人周限额（可选）
- 重复占用限制（可选）：防止反复占同一热门时段（防囤场）。
- 取消截止：距离开始前 X 分钟内不可取消（可选）。
- 违约限制（可选）：爽约达到阈值后禁用预约若干天。
- 场馆员/管理员越权（可配置）：可忽略部分限额执行创建/取消，但任何情况下都不得产生重叠。

预约类型（V1）：
- 个人预约（默认）：只占用 1 个场地。
- 团体预约（教师/社团；推荐）：
  - 同一球馆、同一时间窗口占用多个场地。
  - 一个团体预约包含 N 条子预约（每个场地一条）。
  - 参与人名单可选（V1 可不强依赖参与人数据）。
  - 团体预约受球馆规则控制：是否启用、最大场地数、最大时长。

预约状态：
- CREATED（可选内部态）
- CONFIRMED
- CANCELLED
- CHECKED_IN
- NO_SHOW（超时未到由场馆员/管理员标记）

状态流转（V1）：
- CONFIRMED -> CANCELLED（用户按规则取消；场馆员/管理员可强制取消）
- CONFIRMED -> CHECKED_IN（在核销窗口内由场馆员核销）
- CONFIRMED -> NO_SHOW（超过宽限期由场馆员/管理员标记）
- CHECKED_IN / CANCELLED / NO_SHOW 为终态

### 5.5 取消预约（小程序端）
- 用户在允许取消的窗口内可取消。
- 取消后立即释放占用时段。
- 团体预约取消需要级联取消所有子预约。
- 超过取消截止时间（开始前 X 分钟）禁止取消。

### 5.6 二维码核销签到（场馆员）
- 小程序为 CONFIRMED 预约生成短时效 QR token。
- 预约详情页提供“获取核销码”接口。
- 场馆员扫码后，后端校验：
  - 预约存在且状态为 CONFIRMED
  - 当前时间处于核销窗口（例如开始前 15 分钟至结束）
  - token 签名与过期时间
- 校验通过：状态变更为 CHECKED_IN，并记录核销人、核销时间。
- 核销记录需落库，保证可追溯。
- 手动核销兜底（可选）：按 `booking_no` 查询并手动核销。

### 5.7 违约与限制（V1 可选，推荐做）
- 场馆员可在宽限期后标记 NO_SHOW。
- 系统记录违约并执行限制策略。

建议限制策略：
- 近 30 天 NO_SHOW >= 3 次 -> 禁用预约 7 天。
- 管理员可清除违约记录/解除限制。

### 5.8 通知（V1 可选）
- 微信订阅消息模板：
  - 预约成功
  - 取消成功
  - 开始前提醒（可选）

### 5.9 后台管理端模块
- 仪表盘：今日预约数、各球馆利用率。
- 预约管理：查询、管理员取消、标记爽约、导出（CSV）。
- 球馆/场地管理。
- 规则管理。
- 用户与角色管理。
- 审计日志。

场馆员常用页面（推荐）：
- 今日排场（按球馆/按场地）
- 扫码核销页
- 爽约标记页

## 6. 数据可见性规则
- 普通用户只能查看自己的预约详情。
- 普通用户查看可用性：不展示其他人的身份信息，只显示 occupied/free。
- 场馆员：仅可查看其被分配球馆的预约人姓名/电话等必要信息。
- 管理员：可查看全部。

## 7. 非功能需求
### 7.1 性能
- 可用性查询（单场地/单日）在常规校园负载下应 <= 300ms。
- 创建预约必须具备并发安全性，避免双重占用。

### 7.2 安全
- API 认证：JWT；刷新策略可选。
- RBAC：所有后台/场馆员接口必须鉴权与授权。
- 限流：登录与创建预约接口可选。

### 7.3 可靠性与审计
- 所有预约状态变更必须进入审计日志。
- 核销必须幂等（重复扫码不会产生重复核销记录）。

### 7.4 兼容性
- 小程序：覆盖主流机型与微信基础库。
- 后台：现代浏览器（Chrome/Edge）。

## 8. 可配置参数（建议默认）
- `slot_minutes`：60。
- `book_ahead_days`：7。
- `cancel_cutoff_minutes`：30（开始前 30 分钟不可取消）。
- 核销窗口：开始前 15 分钟至结束（或开始后 15 分钟内）。
- 爽约宽限：开始后 15 分钟。
- 个人日限额：2 个时段/天（示例，落地时按学校调整）。

## 9. 验收标准（V1）
- 并发创建预约不出现双重占用。
- 可用性页面对预约变更在 1 秒内可见（默认强一致；若采用最终一致需显式说明）。
- 二维码核销幂等：重复扫码不会重复写入。
- 取消后立即释放时段。
- 后台/场馆员接口具备严格的角色权限控制。

## 10. 错误码约定（前后端对齐）
- `SLOT_CONFLICT`：时段冲突
- `OUT_OF_OPEN_HOURS`：不在营业时间
- `OUT_OF_BOOKING_WINDOW`：超出可预约窗口
- `LIMIT_EXCEEDED`：超过个人限额
- `CANCEL_NOT_ALLOWED`：已超过取消截止时间
- `CHECKIN_NOT_ALLOWED`：不在核销窗口
- `INVALID_REQUEST`：参数非法
- `UNAUTHORIZED`：未登录
- `FORBIDDEN`：无权限
- `INTERNAL_ERROR`：服务内部错误

## 10. 不在范围（V1）
- 支付与退款。
- 跨学校多租户。
- 场地内“座位级”容量管理（除非将场地建模为容量单位）。
## 5.10 智能提醒与一键改期（V1 可选，推荐 V1.5）
### 5.10.1 候补/可用提醒（不自动占用）
- 用户可对某个“场地+日期+时段窗口”发起关注（候补）。
- 当有取消释放或场地从停用恢复导致可用时，触发提醒（订阅消息/站内通知）。
- 系统不自动替用户创建预约，以降低公平性争议。

### 5.10.2 一键改期/改场地
- 对已确认预约，提供“改期/改场地”向导：
  - 先按推荐算法给出可行替代方案。
  - 用户选择后执行“原单释放 + 新单创建”的原子替换（失败则保留原单）。

## 5.11 智能统计（V1 可选，推荐 V2）
- 后台展示：按球馆/星期/时段的占用率热力图。
- 推荐排序可引入历史占用率/成功率权重（数据驱动排序）。
- 智能推荐接口在常规负载下响应 <= 300ms（不含网络）。
- 推荐结果可解释（每条方案包含推荐理由/标签）。
