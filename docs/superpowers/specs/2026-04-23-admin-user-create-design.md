# 管理端新增后台账号与预置用户设计说明

## 背景

当前管理端已经具备“后台账号管理”和“用户管理”两个列表页，但都缺少新增能力：

- 后台账号只能查看与分配球馆，无法创建新的场馆管理员账号
- 普通用户只能查询、查看详情与切换状态，无法由管理员预置待绑定用户

用户已确认本次范围如下：

- 后台账号新增只允许创建“场馆管理员”
- 普通用户新增采用“预置待绑定”模式，不做直接建档

## 目标

在保持现有页面结构和权限模型基本不变的前提下，为管理端补齐两条最小可用的新增路径：

1. 在“后台账号管理”页新增“新增场馆管理员”
2. 在“用户管理”页新增“新增预置用户”

## 设计原则

1. 最小改动
   - 复用现有列表页，不新增独立路由页
   - 前端采用弹窗表单，后端只补最小创建接口

2. 保持现有权限边界
   - 仅 `ADMIN` 可以执行新增操作
   - 后台账号创建后固定为 `VENUE_STAFF`
   - 普通用户创建后固定为“未绑定”

3. 保持既有业务语义
   - 场馆管理员必须有球馆范围，否则登录会被拒绝
   - 预置用户不能伪造微信身份，不写入 `openid`

## 方案说明

### 一、后台账号新增

在 `后台账号管理` 页面增加按钮与弹窗，录入：

- 用户名
- 姓名
- 手机号
- 邮箱
- 初始密码
- 可管理球馆（多选，必填）

后端新增 `POST /admin/admin-users`：

- 新建 `admin_user`
- 写入 `admin_user_role` 中的 `VENUE_STAFF`
- 写入 `venue_staff` 绑定关系
- `status` 默认 `1`
- 密码按现有后台登录方式进行哈希存储

校验规则：

- 用户名不能为空且不可重复
- 初始密码不能为空
- 至少分配一个球馆
- 球馆 ID 必须全部有效

### 二、普通用户新增

在 `用户管理` 页面增加按钮与弹窗，录入：

- 姓名
- 手机号
- 工号/学号
- 用户类型

后端新增 `POST /admin/users`：

- 创建一条待绑定用户记录
- `openid`、`union_id` 为空
- `is_bound = 0`
- `status = 1`
- `no_show_count = 0`

校验规则：

- 工号/学号不能为空且不能与现有记录重复
- 若填写手机号，则不能与现有未删除用户重复
- 用户类型必填

### 三、兼容性修正

本次在普通用户列表页顺手修正两处已知契约问题：

1. 搜索参数改为后端已支持的 `keyword + status`
2. 用户预约记录、违约记录列表改为兼容后端当前返回的数组结构

## 影响范围

### 后端

- `AdminUserAdminController`
- `AdminUserManagementService`
- `UserAdminController`
- `UserService`
- 新增创建请求 DTO
- 可能补充 `AdminUserRepository`、`UserRepository` 查询方法

### 前端

- `admin-web/src/views/admin-user/list.vue`
- `admin-web/src/views/user/list.vue`
- `admin-web/src/api/admin-user.js`
- `admin-web/src/api/user.js`

## 验证要求

1. 后台账号新增成功后，列表中立即可见，角色显示为“场馆管理员”
2. 新账号使用初始密码可登录后台，并只能访问已分配球馆的数据
3. 预置用户新增成功后，在用户列表可见，绑定状态为“未绑定”
4. 普通用户页搜索、预约记录、违约记录在本次改动后保持正常
