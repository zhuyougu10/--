# 场馆管理员角色设计说明

## 目标

在现有后台管理体系中新增“场馆管理员”能力，但不新增新的角色码，直接复用已有角色码 `VENUE_STAFF`，并统一前后端中文文案为“场馆管理员”。

该角色只负责自己被分配的球馆，支持管理员与球馆多对多绑定。场馆管理员可以管理自己名下球馆的场地、时段、预约、核销，但不能管理后台账号，也不能访问全局统计。

## 现状

### 已有基础

- 数据库已有角色表 `role`，其中已存在 `VENUE_STAFF`
- 数据库已有管理员与球馆多对多关系表 `venue_staff(admin_user_id, venue_id)`
- 路由层已允许 `/admin/**` 由 `ADMIN` 和 `VENUE_STAFF` 访问

### 当前缺口

- 后台登录时固定签发 `ADMIN`，导致 `VENUE_STAFF` 实际未进入运行态
- 权限切面对后台账号存在直通逻辑，导致细粒度权限基本未生效
- 球馆、场地、预约、核销等服务层未按“当前管理员可管理球馆”做数据范围收口
- 管理端未根据角色隐藏后台账号管理、全局统计等无权限入口

## 设计原则

### 1. 复用现有模型，避免新增平行概念

本次不新增 `VENUE_ADMIN` 等新角色码，直接复用 `VENUE_STAFF`。数据库已有角色和关联表，继续沿用可以把改动集中在权限真正生效上，而不是做额外的数据迁移和兼容处理。

### 2. 权限判断与数据范围同时生效

仅有角色判断不够，必须同时做两层收口：

- 权限点校验：决定“能不能做这类操作”
- 数据范围校验：决定“能操作哪些球馆的数据”

只有两层同时存在，场馆管理员才能真正被限制在自己负责的球馆范围内。

### 3. 前端限制只做减误操作，后端限制才是最终兜底

管理端会隐藏无权限菜单、限制球馆筛选范围，但真正的安全边界以服务端校验为准。

## 角色与能力边界

### 超级管理员 `ADMIN`

- 保持现有全局管理能力
- 可访问全部球馆、场地、预约、核销、后台账号、统计数据

### 场馆管理员 `VENUE_STAFF`

- 仅可访问自己被分配的球馆
- 可管理自己球馆的：
  - 球馆信息编辑
  - 场地
  - 营业时段与可预约时段
  - 预约记录
  - 核销操作与核销记录
- 不可管理：
  - 后台账号
  - 全局统计
  - 非授权球馆的数据
  - 全局用户管理
  - 新建/删除球馆

## 数据模型设计

### 复用现有关联表

继续使用 `venue_staff(admin_user_id, venue_id)` 表表达管理员与球馆的多对多关系。

该表语义定义如下：

- 一名场馆管理员可以关联多个球馆
- 一个球馆可以关联多个场馆管理员
- `ADMIN` 不依赖该表做数据范围判断，默认拥有全局访问权
- `VENUE_STAFF` 的数据范围完全由该表决定

`venue_staff` 需要保证 `(admin_user_id, venue_id)` 组合唯一，避免重复绑定导致查询结果重复、批量保存不幂等。若当前数据库尚无唯一约束，则通过新增迁移补齐；若历史数据存在重复记录，则迁移中先做去重再加约束。

### 角色文案映射

保留数据库角色码 `VENUE_STAFF` 作为机器可读值，不修改既有鉴权、路由、接口字段中的角色码语义。

仅在展示层把 `VENUE_STAFF` 映射为中文“场馆管理员”。

## 后端设计

### 1. 登录签发

后台登录不再固定签发 `ADMIN`，而是根据管理员账号实际角色签发 token。

预期行为：

- 若账号拥有 `ADMIN`，则按超级管理员身份登录
- 若账号拥有 `VENUE_STAFF`，则按场馆管理员身份登录
- 若账号既没有 `ADMIN` 也没有 `VENUE_STAFF`，则拒绝后台登录

若账号同时拥有多个后台角色，则优先级为：

- `ADMIN` 优先于 `VENUE_STAFF`

这样可以避免同一账号在后台上下文中出现不确定角色。

### 2. 权限校验

后台权限校验保持现有 `@RequirePermission` 机制，但修正“后台账号默认直通”的逻辑。

目标行为：

- `ADMIN` 允许全局通过
- `VENUE_STAFF` 必须走真实权限点校验

直通判断必须基于当前 token 或会话中的有效角色，而不是基于“后台管理员身份”本身。也就是说，只有角色为 `ADMIN` 时才允许直通；后台账号如果角色为 `VENUE_STAFF`，即使也是后台用户，也不能绕过权限点判断。

`VENUE_STAFF` 建议保留或补齐以下权限：

- `venue:read`
- `venue:update`
- `court:read`
- `court:create`
- `court:update`
- `court:delete`
- `booking:read`
- `booking:cancel`
- `booking:checkin`
- `booking:mark_no_show`
- 与营业时段、可预约时段相关的读取和编辑权限

如果现有初始化权限缺少部分写权限，则通过新增迁移补齐，不修改历史迁移文件。

若当前项目的时段能力没有单独权限点，则需要在实现中明确复用到现有球馆或场地管理权限，避免出现“业务允许但接口权限缺失”的空档。

### 3. 数据范围服务

新增统一的“当前管理员可管理球馆范围”查询能力，供多个业务服务复用。

建议提供以下能力：

- 获取当前管理员身份与角色
- 获取当前管理员可管理球馆 ID 集合
- 判断某个 `venueId` 是否在当前管理员可管理范围内
- 判断某个 `bookingId`、`courtId` 对应球馆是否在范围内

该能力建议沉淀为独立服务或查询组件，避免在各业务服务里重复拼装同样的逻辑。

需要明确零授权场景：若 `VENUE_STAFF` 没有关联任何球馆，则列表接口返回空集，详情、编辑、删除、核销、预约管理等定向操作一律拒绝，不能因为授权集合为空而退化为全量可见。

### 4. 业务收口点

#### 球馆管理

- 列表：
  - `ADMIN` 看全部
  - `VENUE_STAFF` 只看自己关联球馆
- 详情：
  - 非授权球馆禁止访问
- 编辑：
  - 仅允许编辑自己关联球馆
- 新建/删除：
  - 仅 `ADMIN` 可执行

#### 场地管理

- 列表：仅返回当前管理员可管理球馆下的场地
- 新增：仅允许在自己关联球馆下创建场地
- 更新/删除：仅允许操作自己球馆的场地

#### 预约管理

- 列表分页：自动限制为当前管理员可管理球馆
- 今日排场：自动限制为当前管理员可管理球馆
- 场馆预约查询：仅允许查询授权球馆
- 取消预约、标记违约等管理动作：仅允许操作授权球馆下的预约

#### 核销管理

- 核销记录：仅返回授权球馆下的记录
- 扫码核销/手动核销：仅允许核销授权球馆下的预约

#### 用户管理与全局统计

- `VENUE_STAFF` 不允许访问
- 对应接口保留现有权限点或收紧为仅 `ADMIN`

### 5. 统一收口规则

凡是通过 `venueId`、`courtId`、`bookingId` 等间接定位到球馆的数据读写入口，都必须先统一换算出所属球馆，再经过同一套授权范围判断。不能只在列表查询做收口，而遗漏详情查询、按 ID 更新、删除、核销、取消预约、标记违约等非列表型入口。

这条规则同样适用于独立的违约管理服务入口，例如通过预约单号间接定位球馆的 `markNoShow` 一类操作。

## 管理端设计

### 1. 角色展示

- 角色码 `VENUE_STAFF` 在前端统一显示为“场馆管理员”

### 2. 菜单与路由

- 场馆管理员隐藏：
  - 仪表盘/全局统计入口
  - 用户管理
  - 后台账号管理
  - 全局统计
- 保留：
  - 球馆管理
  - 场地管理
  - 预约管理
  - 核销管理

### 3. 页面行为

- 球馆列表只显示当前可管理球馆
- 若存在球馆筛选，下拉框只展示已授权球馆
- 球馆管理页中 `ADMIN` 专属操作按钮需要隐藏或禁用，例如“新建球馆”“删除球馆”
- 如果场馆管理员尝试直接访问无权限页面，前端可做跳转或提示，但不依赖前端做安全判断

## 错误处理

对 `VENUE_STAFF` 的越权访问统一返回明确错误：

- 若目标资源存在但不在授权范围内，返回无权限错误
- 若请求的是后台账号管理、统计等明确禁区，返回权限不足
- 若管理员未分配任何球馆，则相关列表返回空集，相关写操作返回权限不足

错误文案保持中文，便于后台使用人员理解。

## 测试设计

至少覆盖以下场景：

1. `ADMIN` 登录后仍可访问全量球馆数据
2. `VENUE_STAFF` 登录后仅能看到已授权球馆
3. `VENUE_STAFF` 访问非授权球馆详情失败
4. `VENUE_STAFF` 编辑非授权球馆失败
5. `VENUE_STAFF` 在授权球馆下新增/编辑/删除场地成功
6. `VENUE_STAFF` 查看预约列表时只返回授权球馆数据
7. `VENUE_STAFF` 扫码核销非授权球馆预约失败
8. `VENUE_STAFF` 无法访问后台账号管理接口
9. `VENUE_STAFF` 无法访问统计接口
10. 一个管理员绑定多个球馆时，可同时管理多个授权球馆
11. 一个球馆绑定多个管理员时，多个管理员都可正常访问该球馆数据
12. `VENUE_STAFF` 未分配任何球馆时，列表返回空，定向操作均被拒绝
13. 详情查询、按 ID 更新、删除、核销、取消预约等非列表入口同样被授权范围拦截
14. `VENUE_STAFF` 无法访问仪表盘和用户管理页面
15. `VENUE_STAFF` 看不到球馆管理中的“新建球馆”“删除球馆”等 `ADMIN` 专属操作

## 兼容性与迁移策略

### 兼容原则

- 不修改历史迁移文件
- 尽量复用现有表和权限模型
- 对超级管理员现有能力保持兼容

### 迁移策略

若发现 `VENUE_STAFF` 现有权限不足，则新增迁移脚本补充 `role_permission` 数据。

后台必须提供“为场馆管理员分配球馆”的可运营入口，不能把这一步留给人工改库。

建议由 `ADMIN` 在后台账号管理页中维护管理员与球馆的多对多绑定关系，并提供以下能力：

- 查看某后台账号已绑定球馆
- 为后台账号新增或移除球馆绑定
- 保存后即时影响该账号的数据范围

这里的“即时生效”定义为：后端每次请求实时按当前数据库绑定关系计算授权范围，不依赖长生命周期权限缓存。若当前登录态或前端菜单存在缓存，则在绑定保存后要求目标账号重新获取 profile 或重新登录，以刷新可见菜单与前端状态；服务端数据权限以最新数据库关系为准。

若当前账号管理页改造成本过高，则至少需要提供一组仅 `ADMIN` 可用的管理接口，并把这条链路纳入本次验收范围。

## 非目标

本次不处理以下内容：

- 新增更细粒度角色体系
- 重构整个权限系统
- 设计独立的球馆管理员工作台
- 修改小程序端角色体系

## 建议实施顺序

1. 修正后台登录角色签发
2. 修正后台权限切面直通逻辑
3. 建立管理员可管理球馆范围查询能力
4. 逐步收口球馆、场地、预约、核销服务
5. 收紧后台账号与统计访问
6. 管理端按角色隐藏菜单与限制筛选
7. 补充测试与验证
